### 2026-04-01: 記録タブHeroの3カラムは表示値を直接アニメーションする

- レポートタブのKPI行は `onAppear + spring(delay)` で 0 から最終値へ遷移する挙動を基準とする
- 記録タブはHero自体を後から挿入せず、0表示のまま先にマウントする
- Hero用データは個別プロパティを順番に更新せず、`RecordingHeroMetrics?` を `nil` から1回だけ確定代入する
- `TodayHeroView` はその確定値を受けて 1 回だけ `0 -> 最終値` を同じ spring / delay で流す
- 時間KPIの描画コードはレポートタブの `kpiItem` と同一にし、記録タブではラベルだけ初期Y位置と透明度を持たせて見え方を合わせる
- 記録タブの時間KPIは `100h` 以上の値だけ compact 表示に切り替え、`xxxh` として1行維持する。レポートタブのKPI演出・描画は元実装のまま維持する
- 桁位置補正のための固定スロットや幅予約のような対症療法は採用しない
- レポートタブの実装・演出は変更しない

### 2026-03-28: レビューチケット制度設計（v2.1）

- **チケット制の相互レビュー**
  - レビューを1件書く → チケット1枚獲得
  - チケット1枚消費 → 依頼受付ONのユーザーにレビュー依頼可能
  - 新規ユーザーに初回1枚無料チケット付与（有効期限1ヶ月）。以降のチケットは期限なし
- **レビュー形式**: App Store式（5段階星評価 必須 + テキストコメント 任意）
- **フロー**: 依頼送信 → レビュアーが星評価を送信 → 自動完了 → チケット即付与
- **回答期限**: 3日。放置で信頼スコアが低下
- **拒否**: 依頼受付ONにしている以上、拒否不可。放置すると信頼スコア低下
- **依頼受付設定**: オンボーディングで設定。後から設定画面で変更可能
- **完了レビューはプロジェクトページに公開**。開発者からの返信可能
- **悪意あるレビュー**: 報告機能で対応
- **プロフィール表示**: レビュー実績数 + 信頼スコア
- **参考**: Critique Circle / Scribophile のクレジット型レビュー交換モデル

### 2026-03-28: カテゴリ・カラーシステム設計

- **2層構造**: カテゴリ(親) → プロジェクト(子)。色はカテゴリが持ち、プロジェクトは親の色を継承
- **標準カテゴリ7個**(全ユーザー共通ブランドカラー): 開発, デザイン, 学習, ミーティング, ライティング, マーケティング, 事務
- **カスタムカテゴリ**: ユーザーが自由に追加。5色のカスタムパレットから選択
- **合計12色パレット**: 標準7色 + カスタム5色。Asset Catalogでライト/ダーク両対応
- **色だけで区別する**(アイコン不使用)。ソーシャルフィードで他人のレポートを見た時「この色=開発」と共通認識になる
- **プロジェクト単位で各色別表示**。同カテゴリ内の濃淡方式は不採用（区別しにくいため）
- **クロスプラットフォーム**: 色はHEX値でSupabaseに保存。iOS/Android/Web共通
- 参考: Toggl Track(14色パレット), Apple Screen Time(カテゴリ固定色), WakaTime(23活動タイプ)

### 2026-03-31: マイプロダクト登録の導線設計

- **マイプロダクト = ポートフォリオのプロジェクト**。記録タグのプロジェクト名と同じエンティティ
- 登録導線は3つ:
  1. **オンボーディング** — 初回セットアップで「あなたが運用しているサービスを登録」
  2. **記録タブ初回訪問時** — サービス未登録なら登録を促すUI
  3. **記録タブ内で随時** — タグ作成ウィザードのstep 3で「新しいサービスを追加」（実装済み）
- 登録したサービスはプロフィールの「マイプロダクト」セクションにも表示される
- 記録時は常設の時間ピッカーで時間を選び、タグ（カテゴリ+サービス）を選択して記録する。タグ未選択なら「その他」で保存

### 2026-04-04: 記録画面の時間入力は既存の常設ループピッカーを維持

- **常設ホイールは維持**。記録画面上に常時表示する
- **実装**: `DurationPicker`（`UIPickerView` ベースの既存ループピッカー）をそのまま使う
- **理由**: `UIDatePicker(.countDownTimer)` への差し替えで見た目が変わり、既存の質感を損ねたため。ユーザー評価の高かった常設ピッカーを基準に戻す
- **レイアウト**: `TimePickerSection` 全体は持ち上げず、固定高さの表示窓の中でホイール数字列だけを上へ寄せる。ボタン行やタグ行は動かさない
- **補足**: 横線に見えていたものはピッカーではなく記録Heroのカテゴリ内訳バーだったため、記録画面では表示しない
- **影響範囲**: 記録タブの時間入力 UI のみ。Hero アニメーションとレポートタブは変更しない

- iOS 26 / Swift 6.2 の最新APIのみ使用。古い書き方（ObservableObject, NavigationView, @StateObject等）は全面禁止
- 対象ユーザーはエンジニアなので新しいデバイス率が高く、影響は限定的と判断
- 2026年3月時点でiOS 26普及率は世界約76%、日本は推定80%前後
- 詳細ルールは `.claude/rules/architecture.md` の「iOS 26最低ターゲット」セクションに記載

### 2026-03-31: UserProfileViewをProfileViewと統一

- UserProfileViewの旧デザイン（4バッジカード、スキルセクション、横スクロールプロジェクトカード）を廃止
- ProfileViewと同じ構成に統一: コンパクトな累計/連続テキスト、縦リストマイプロダクト（レビュー付き）、週間チャート
- **メッセージ機能は将来実装**（国の規制により現時点で実装不可）。UIからボタンを削除
- **他人のプロフィールではいいね/ブックマークタブを非表示**。投稿のみ表示。自分のプロフィールのみ3タブ（投稿/いいね/ブックマーク）

### 2026-04-04: 遅延登録オンボーディング（Delayed Registration）

- ログインをステップ5に移動。ステップ1-4はログイン不要で体験させる
- Duolingoモデル: 遅延登録導入でリテンション12%→55%（+20%）。離脱率-56%
- フロー: ウェルカム → 技術タグ選択 → 初回記録体験 → ログイン（データ保存動機）→ ハンドル設定
- ウェルカム画面に「アカウントをお持ちの方」リンクで既存ユーザー即ログイン可能
- ステップ1-4の入力データはメモリ保持、ログイン後にサーバー同期
- 全体60秒以内、全ステップスキップ可能（ログイン以外）

### 2026-04-06: オンボーディング全面リデザイン (9画面プロダクト登録フロー)

- 9画面は意図的。離脱防止は画面数削減ではなくUX/UI品質で担保
- フロー: welcome → appShowcase → tutorialIntro → platform → language → projectName → saving → accountPrompt → profileSetup
- オンボーディングの主目的は「マイプロダクト (SDProject)」の登録。プロフィールのマイプロダクト + 記録タブのプロジェクト選択に反映
- SDProject に platform / languages フィールド追加
- 認証優先順位: Apple Sign In (最優先) > Google (最優先) >> GitHub / X / メール (後回し)
- Apple: ASAuthorizationAppleIDProvider + Supabase signInWithIdToken (実装済み)
- Google: Supabase OAuth フロー (追加SDK不要、ASWebAuthenticationSession使用)
- 後回し認証: GitHub OAuth / X OAuth / Email+Password は後日対応
- Supabase への書き込みは auth 完了後。オンボーディング中は SwiftData ローカル保存のみ
